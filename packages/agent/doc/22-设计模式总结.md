# 设计模式总结

本文档从"如何构建类似系统"的角度，梳理 DeepAgents 中使用的关键设计模式和值得借鉴的经验。

---

## 1. 工厂函数模式（Factory Function）

**位置**：`createDeepAgent`、`createFilesystemMiddleware`、`createSubAgentMiddleware` 等

所有核心对象都通过工厂函数创建，而非直接 `new`：

```typescript
// 推荐：工厂函数
const agent = createDeepAgent({ ... });
const middleware = createFilesystemMiddleware({ backend });

// 不推荐：直接构造
const agent = new DeepAgent({ ... });  // DeepAgents 不提供这种方式
```

**好处**：
- 工厂函数可以做参数校验（如工具名冲突检测）
- 工厂函数可以返回复杂的联合类型（如 `DeepAgent<TypeConfig>`）
- 工厂函数是 `async` 友好的（如果初始化需要异步操作）

---

## 2. 中间件模式（Middleware Pattern）

**位置**：`src/middleware/`

采用洋葱（Onion）模型，每个中间件可以：
- 在 LLM 调用前后注入逻辑（`wrapModelCall`）
- 在工具执行前后注入逻辑（`wrapToolCall`）
- 在 Agent 启动前执行初始化（`beforeAgent`）
- 注册新工具（`tools`）
- 扩展状态 Schema（`stateSchema`）

```typescript
// 中间件注册顺序决定执行顺序
const middleware = [
  todo,           // ← 最后包裹（最内层）
  skills,
  filesystem,
  subagents,
  summarization,
  patch,
  custom,         // ← 最先包裹（最外层）
];
```

**关键经验**：固定中间件顺序，不让用户排序，保证行为可预期。

---

## 3. 策略模式（Strategy Pattern）

**位置**：`src/backends/`

通过 `BackendProtocolV2` 接口抽象文件操作，实现可替换的存储策略：

```
BackendProtocolV2（接口）
    ├── StateBackend    → LangGraph 状态
    ├── StoreBackend    → LangGraph Store
    ├── FilesystemBackend → 本地磁盘
    ├── CompositeBackend → 路由组合
    └── LangSmithSandbox → 云沙箱
```

**关键经验**：
- 接口只定义行为，不关心实现细节
- 所有操作返回结构体（含 `error` 字段），不抛出异常
- `resolveBackend()` 统一处理版本适配和工厂函数解析

---

## 4. 组合模式（Composite Pattern）

**位置**：`CompositeBackend`

```typescript
new CompositeBackend([
  { prefix: "/projects/", backend: new FilesystemBackend(...) },
  { prefix: "/tmp/",      backend: new StateBackend() },
  { prefix: "/",          backend: new StoreBackend() },  // 兜底
])
```

**关键经验**：组合模式使子系统可以无限嵌套，增加了灵活性而不增加核心接口复杂度。

---

## 5. 类型品牌（Phantom Type / Type Brand）

**位置**：`src/types.ts`

```typescript
type DeepAgent<TTypes extends DeepAgentTypeConfig> = ReactAgent<TTypes> & {
  readonly "~deepAgentTypes": TTypes;  // 类型品牌
};
```

`"~deepAgentTypes"` 永远不会在运行时被访问，只存在于类型层面，用于从实例中"取回"泛型参数：

```typescript
// 从实例中提取泛型类型
type ResolveDeepAgentTypeConfig<T> = T extends { "~deepAgentTypes": infer Types } ? Types : never;
```

**关键经验**：用 `readonly "uniqueKey": TTypes` 作为类型品牌，比用类继承更轻量，且不影响运行时行为。

---

## 6. Reducer 模式（for 并发安全状态）

**位置**：`src/middleware/fs.ts`

```typescript
// 普通 LastValue 语义（会被并发覆盖）
state.files = subagentAFiles;  // ❌ 覆盖了 subagentB 的结果

// Reducer 语义（正确合并）
fileDataReducer(current, update) → 合并后的状态  // ✓
```

**关键经验**：并发子代理共享状态时，使用 Reducer（而非 LastValue）合并各自的更新，确保不丢失任何一方的写入。

---

## 7. 驱逐（Eviction）模式

**位置**：`src/middleware/fs.ts`

当 LLM 上下文接近 token 限制时，自动将大数据"驱逐"到持久化存储，并用指针（文件路径）替换原内容：

```
大工具结果（80KB）
    → 写入 /large_tool_results/{id}
    → 替换为：
      "结果已保存到 /large_tool_results/{id}，
       使用 read_file 分页读取"
```

**关键经验**：驱逐是控制上下文大小的核心手段，LLM 仍然可以通过工具获取完整内容，但上下文窗口保持精简。

---

## 8. 渐进式披露（Progressive Disclosure）

**位置**：`src/middleware/skills.ts`

技能内容分两阶段加载：
1. **摘要阶段**：只加载技能名称和描述（元数据）
2. **使用阶段**：LLM 决定使用某技能时，加载完整 Markdown 内容

```typescript
// 注入摘要（轻量）
systemPrompt += `
## Available Skills
- web-search: Use Tavily to search the web
- code-review: Systematically review code
`;

// 使用时注入完整内容（一次性，特定技能）
systemPrompt += fs.readFile("/skills/web-search/SKILL.md");
```

**关键经验**：避免一次性加载所有扩展内容。先给 LLM 足够选择的信息，按需加载细节。

---

## 9. 错误返回而非异常（Result Pattern）

**位置**：`src/backends/protocol.ts`

```typescript
// 错误通过结构体返回（而非 throw）
interface ReadResult {
  error?: string;    // 失败时有值
  content?: string;  // 成功时有值
}

// 调用方检查 error
const result = await backend.read(path);
if (result.error) {
  return `Error: ${result.error}`;  // 返回错误文本给 LLM
}
return result.content;
```

**关键经验**：工具的错误应该返回给 LLM（让它重试或换策略），而不是抛出异常（中断整个 Agent）。只有配置错误（构建时）才抛异常。

---

## 10. 类型版本化（Versioned Types）

**位置**：`src/backends/v1/protocol.ts`、`src/backends/v2/protocol.ts`

```
BackendProtocolV1 → 已弃用，保留向后兼容
BackendProtocolV2 → 当前标准

adaptBackendProtocol(v1): BackendProtocolV2  // V1 → V2 适配器
```

**关键经验**：用版本后缀标记接口（V1、V2），提供适配器，而非破坏性修改。

---

## 11. 跨 Realm 安全的 instanceof

**位置**：`src/errors.ts`、`src/backends/protocol.ts`

```typescript
// 问题：ESM/CJS 多版本导致 instanceof 失败
error instanceof ConfigurationError  // ❌ 跨 realm 可能失败

// 解决方案：Symbol.for + 静态方法
const SYMBOL = Symbol.for("my.unique.key");

class MyError extends Error {
  [SYMBOL] = true;
  static isInstance(e): e is MyError {
    return typeof e === "object" && e !== null && e[SYMBOL] === true;
  }
}

MyError.isInstance(error)  // ✓ 跨 realm 安全
```

## 构建类似系统的建议

| 方面 | DeepAgents 的做法 | 建议 |
|------|-----------------|------|
| 底层图引擎 | LangGraph（状态机） | 需要状态管理和 checkpointing |
| 中间件机制 | LangChain middleware API | 考虑设计扩展点的标准接口 |
| 文件操作抽象 | BackendProtocolV2 | 从一开始设计接口，再实现后端 |
| 状态合并 | ReducedValue + Reducer | 并发更新用 Reducer，单值用 LastValue |
| 上下文管理 | 驱逐 + 摘要双策略 | 两者都需要，针对不同大小问题 |
| 技能系统 | Markdown + YAML frontmatter | 简单且人类友好，LLM 直接理解 |
| 类型系统 | TypeScript 泛型类型推断 | 类型品牌模式适合需要携带泛型的场景 |
| 错误处理 | 结果对象 + 错误码 | 运行时返回错误，构建时抛出异常 |
