## 硬编码限制

| 约束项             | 值                     | 来源文件                                     |
| ------------------ | ---------------------- | -------------------------------------------- |
| 技能文件最大大小   | 10 MB                  | `skills.ts`: `MAX_SKILL_FILE_SIZE`           |
| 技能名最大长度     | 64 字符                | `skills.ts`: `MAX_SKILL_NAME_LENGTH`         |
| 技能描述最大长度   | 1024 字符              | `skills.ts`: `MAX_SKILL_DESCRIPTION_LENGTH`  |
| 二进制文件最大读取 | 10 MB                  | `fs.ts`: `MAX_BINARY_READ_SIZE_BYTES`        |
| 单行最大字符数     | 5000 字符              | `backends/utils.ts`: `MAX_LINE_LENGTH`       |
| 工具结果驱逐阈值   | 20000 tokens（≈80KB）  | `fs.ts`: `toolTokenLimitBeforeEvict`         |
| 消息驱逐阈值       | 50000 tokens（≈200KB） | `fs.ts`: `humanMessageTokenLimitBeforeEvict` |
| Agent 最大递归步数 | 10000 步               | `agent.ts`: `recursionLimit`                 |
| 字符/token 估算比  | 4 字符/token           | `fs.ts`: `NUM_CHARS_PER_TOKEN`               |
| read_file 默认行数 | 100 行                 | `fs.ts`: `DEFAULT_READ_LINE_LIMIT`           |

## 内置工具名称保留

以下工具名不能被用户自定义工具使用（构建时检测，抛出 `ConfigurationError`）：

```typescript
const BUILTIN_TOOL_NAMES = new Set([
  // 文件系统工具
  "ls", "read_file", "write_file", "edit_file", "glob", "grep", "execute",
  // 子代理工具
  "task",
  // 异步子代理工具
  "create_async_task", "check_async_task", "update_async_task", "cancel_async_task",
  // LangChain 内置
  "write_todos",
]);
```

```
正则：/^[a-z0-9-]+$/

允许：
  web-search         ✓
  code-review-v2     ✓
  data123            ✓

不允许：
  Web_Search         ✗（大写字母，下划线）
  Code Review        ✗（空格）
  my.skill           ✗（点号）
  -leading-dash      ✗（连字符开头）
```

## 子代理状态隔离规则

以下状态字段**不会**传递给子代理：

```typescript
const EXCLUDED_STATE_KEYS = [
  "messages",           // 用任务描述替换
  "todos",              // 无 Reducer，子代理独立管理
  "structuredResponse", // 无 Reducer，子代理独立生成
  "skillsMetadata",     // 子代理独立加载技能
  "memoryContents",     // 子代理独立加载记忆
];
```

其他字段（如 `files`）会传递，使子代理能访问主 Agent 的文件。

## 后端 V1 弃用

V1 后端协议已标记为 `@deprecated`：

```typescript
/** @deprecated Use BackendProtocolV2 instead */
export interface BackendProtocol extends BackendProtocolV1 {}

/** @deprecated Use SandboxBackendProtocolV2 instead */
export interface SandboxBackendProtocol extends SandboxBackendProtocolV1 {}

/** @deprecated Pass a pre-constructed backend instance instead of a factory */
export type BackendFactory = (runtime: BackendRuntime) => ...;
```

旧接口仍受支持（通过 `adaptBackendProtocol` 适配），但新代码应使用 V2。

## 已知行为边界

### read_file 的行限制行为

- 超长行（> 5000 字符）被分割为续行（如 `5.1`、`5.2`）
- 续行计入 `limit` 限制（即实际读取的原始行可能少于 `limit`）
- 二进制文件无法分页（`offset`/`limit` 对二进制文件无效）

### grep 搜索行为

- `grep` 使用字面量字符串匹配（不是正则），但 `FilesystemBackend` 的底层实现可能支持正则
- 特殊字符（`(`、`)`、`|` 等）被视为字面量
- 结果数量有截断（超过 200 行时截断）

### StateBackend 的局限

- 文件仅在同一 LangGraph 线程内可见
- 没有 `checkpointer` 时，重新调用 `invoke` 后文件丢失
- 不支持真正的 `execute`（Shell 命令执行）

### 并发写入冲突

- 同一个文件被多个子代理并发写入时，最后写入者获胜（无合并语义）
- `fileDataReducer` 只合并**不同文件键**的更新，不合并同一文件的内容

### Anthropic 提示缓存的限制

- 提示缓存仅对 Anthropic 模型自动启用
- 需要满足 Anthropic 的最小 token 数要求（`minMessagesToCache: 1`）
- 动态变化的系统提示（如含时间戳）会导致缓存失效

## 中间件执行顺序的隐含约束

中间件注册顺序决定执行顺序，改变顺序可能导致功能异常：

| 如果颠倒...               | 后果                                             |
| ------------------------- | ------------------------------------------------ |
| skills 在 filesystem 之后 | 技能文件在文件系统工具就绪前尝试读取（可能失败） |
| memory 在 cache 之前      | 记忆内容不会被提示缓存优化                       |
| hitl 不在最后             | 人工审批中断可能与其他中间件状态冲突             |

这也是为什么 `createUniverseAgent` 强制使用固定的中间件顺序，而不允许用户重排序。
